**작성일**: 2026-02-05

## 1. 검토 개요

### 1.1 목적

「카고플랜 자체 구축 가능 기능 정밀 분석 v2.0」(이하 "원문 보고서")이 16개 기능의 자체 구축 유효성을 3개 Tier로 분류하고, 3단계 로드맵을 제시한 것에 대해 **비용, 법적/규제, 논리적 정합성, 타 문서 일관성** 관점에서 체계적으로 반박한다.

### 1.2 검토 대상 문서

| 문서 | 역할 |
|------|------|
| 카고플랜 자체 구축 가능 기능 정밀 분석 v2.0 | 반박 대상 (원문 보고서) |
| 해상운임 시장 분석 보고서 v2.0 | 교차 검증 |
| 카고플랜 플랫폼 활용 분석 보고서 v2.0 | 교차 검증 |

### 1.3 검토 방법론

- 원문 보고서의 구체적 문장, 테이블, 수치를 인용하여 반박 근거 제시

## 2. 핵심 지적 요약 (Executive Summary)

원문 보고서는 서프컴퍼니 16개 기능을 분석하여 Tier 1~3으로 분류하고 Phase 1~3 로드맵을 제시한 점에서 유용한 분석 프레임을 제공한다. 그러나 다음 **5가지 범주의 중대한 결함**이 있어, 이 보고서를 기반으로 의사결정을 내리기에는 부족하다.

| # | 결함 범주 | 심각도 | 핵심 문제 |
|---|----------|:------:|----------|
| 1 | **비용 분석 전면 부재** | 🔴 치명적 | 16개 기능 분석에 단 한 건의 비용 추정 없음. 투자 의사결정 불가 |
| 2 | **법적/규제 검토 전면 누락** | 🔴 치명적 | 개인정보보호법, 정보통신망법, 전자상거래법 등 필수 법적 요구사항 미검토 |
| 3 | **논리적 편향** | 🟡 중대 | "자체 구축"을 전제한 분석. Build vs Buy 대안 검토 부재 |
| 4 | **문서 구조/정합성 문제** | 🟠 보통 | "5단계 평가" 표기 오류, Tier-Phase 매핑 불일치, FTE 합산 비현실성 |
| 5 | **타 문서 정합성 문제** | 🟡 중대 | 시장 보고서 권고 무시, 활용 보고서 전략과 충돌 |

## 3. 비용 분석 부재에 대한 반박

### 3.1 개발 비용 추정 부재

**원문 인용**: 원문 보고서는 16개 기능을 분석하고 12개를 상세 평가하면서, "구축 난이도"를 낮음/중간/높음으로만 기술했다(원문 §자체 구축 유효성 분석, 평가 기준 테이블). **구축에 필요한 개발 비용(인건비, 외주비)에 대한 추정은 단 한 건도 없다.**

**반박 근거**: 소프트웨어 개발 비용은 의사결정의 가장 기본적인 변수이다. "구축 난이도"가 낮다는 것이 곧 비용이 적다는 의미가 아니다.

**업계 기준에 의한 예비 추정** (한국소프트웨어산업협회 SW사업 대가산정 가이드 참조):

| Phase | 주요 기능 | 예상 개발 규모 (추정) | 비고 |
|-------|----------|---------------------|------|
| Phase 1 | 로그인, 트래킹, 부킹, 알림, 선박알림 | 중급 웹 애플리케이션 | OAuth 연동, 다수 선사 API 연동, 실시간 알림 등 포함 |
| Phase 2 | 운임지수, 물동량, 블로그, 뉴스레터, 대시보드 | 데이터 수집/시각화 + CMS | 크롤러, 공공데이터 연동, CMS 등 포함 |
| Phase 3 | 해상운임 조회, 데이터맵, 다국어 | 고급 데이터 플랫폼 | 다중 외부 API, 복잡한 시각화 |

원문 보고서에 이러한 추정이 없다는 것은, **"이 기능을 만들 것인가"라는 질문에 대해 가장 중요한 변수를 무시한 것**이다.

### 3.2 인프라/운영 비용 부재

**원문 인용**: 원문 보고서는 "핵심 데이터 소스 정리" 테이블(원문 §핵심 데이터 소스 정리)에서 선사 API를 "무료~유료", 공공데이터를 "공공데이터"로만 기술했다. **서버, 클라우드, DB, CDN, SSL, 모니터링 도구 등 인프라 비용은 일체 언급되지 않았다.**

**반박 근거**: 자체 구축은 개발 후에도 지속적인 인프라 비용이 발생한다.

| 인프라 항목 | 월간 비용 범위 (추정) | 비고 |
|------------|---------------------|------|
| 클라우드 서버 (AWS/GCP) | 수십만~수백만 원 | 트래픽, 스토리지 규모에 따라 |
| DB 서버 (RDS 등) | 수십만 원~ | 데이터 규모에 따라 |
| CDN | 수만~수십만 원 | 정적 자산 규모에 따라 |
| SSL 인증서 | 무료~수십만 원/년 | Let's Encrypt 또는 유료 |
| 모니터링/로깅 | 수만~수십만 원 | Datadog, Sentry 등 |
| 이메일 발송 서비스 | 수만~수십만 원 | Mailchimp, SES 등 (뉴스레터) |
| 알림 서비스 | 건당 과금 | 카카오 알림톡 등 |

이 항목들이 전혀 고려되지 않은 상태에서 "자체 구축 추천"이라는 결론은 근거가 불충분하다.

### 3.3 인건비 실산정 불완전

**원문 인용**: 원문 보고서는 기능별 운영 인력을 FTE(Full-Time Equivalent)로 제시했다. 예: "IT 담당 0.2명", "운영 담당 0.3명" (원문 §운영 종합 매트릭스).

| Phase 1 기능 | 운영 FTE |
|-------------|---------|
| 소셜/이메일 로그인 | 0.2 |
| 카고 트래킹 | 0.8 |
| 선박 입출항 알림 | 0.5 |
| Booking 관리 | 1.3 |
| 실시간 알림 | 0.2 |
| **Phase 1 합계** | **3.0** |

**반박 근거**:

1. **연봉/단가 미반영**: FTE 3.0명이라는 수치만으로는 비용을 알 수 없다. IT 개발자 평균 연봉(한국소프트웨어산업협회 2024년 기준)과 운영 담당자 인건비를 적용해야 실제 비용이 산출된다.
2. **개발 인력과 운영 인력 혼동**: 위 FTE는 "운영 인력"이다. 구축 단계의 개발 인력은 별도로 필요하나 전혀 산정되지 않았다.
3. **중복 고려 부재**: 원문이 "운영 FTE 합산 부재"를 인지하지 못했다. 0.2명, 0.3명 단위의 FTE가 실무에서 어떻게 배분되는지 현실성 검토가 없다.

### 3.4 기존 솔루션 비용 비교 (Build vs Buy) 부재

**원문 인용**: 원문 보고서는 "ShipGo Plug-in vs 자체 구축 전략" 섹션(원문 §ShipGo Plug-in vs 자체 구축 전략)에서 "ShipGo Plug-in으로 고객 니즈를 검증한 뒤, 자체 구축으로 전환"이라는 전략을 제시했다. 그러나 ShipGo Plug-in의 비용(현재 한시적 무료, 유료 전환 시 가격)과 자체 구축 비용의 **정량적 비교는 없다**.

**반박 근거**: 플랫폼 활용 분석 보고서(§1.1)에 따르면 트레드링스의 서비스 비용 구조는 다음과 같다:

| 서비스 | 비용 |
|--------|------|
| LinGo | 무료 |
| ShipGo Plug-in | 한시적 무료 |
| ShipGo | 유료 |
| ZimGo | 유료 |

**자체 구축의 총비용(TCO)이 이러한 외부 솔루션의 비용보다 낮다는 근거가 제시되지 않은 상태에서, 자체 구축을 "추천"한다는 것은 논리적으로 불완전하다.**

### 3.5 기회비용 미산정

**원문 인용**: 원문 보고서는 Phase 1~3 로드맵을 제시하면서(원문 §종합 추천: 카고플랜 자체 구축 로드맵), 개발 리소스가 자체 구축에 투입되는 동안 **포기해야 하는 다른 투자 기회**를 검토하지 않았다.

**반박 근거**: 카고플랜은 포워더이다. 개발 인력과 예산을 자체 구축에 투입하면 다음을 포기하게 된다:

- 기존 고객 영업 강화 (KT&G, 명보 등 대기업 관계 심화)
- 위험물/프로젝트 화물 전문성 강화 (카고플랜의 핵심 차별화 요소)
- 신규 항로/고객 개척
- 기존 시스템 개선

해상운임 시장 분석 보고서(§7.3)도 "A. 자체 플랫폼 구축"의 단점으로 "높은 비용, 긴 개발 기간, 높은 리스크"를 명시했다. 이러한 기회비용을 정량화하지 않으면 투자 의사결정의 합리성을 검증할 수 없다.

### 3.6 ROI 분석 부재

**원문 인용**: 해상운임 시장 분석 보고서(§5.1 필수 검토 항목)는 비용-편익 분석 프레임워크의 4개 항목을 제시했다:

| 항목 | 추정 범위 |
|------|-----------|
| 초기 개발 비용 | **추가 조사 필요** |
| 운영 비용 | **추가 조사 필요** |
| 손익분기점 | **추가 조사 필요** |
| ROI 기간 | **추가 조사 필요** |

**반박 근거**: 시장 보고서가 "추가 조사 필요"라고 명시한 4개 항목 전부가, 자체구축 보고서에서도 **여전히 미해결 상태**이다. 비용-편익 분석 없이 "즉시 착수"(Phase 1)를 권고하는 것은 시장 보고서의 권고를 무시한 것이다.

## 4. 법적/규제 검토 누락에 대한 반박

### 4.1 개인정보보호법 준수 요구사항

**원문 인용**: 원문 보고서는 소셜/이메일 로그인을 Tier 1 "필수 인프라"로 분류하면서(원문 §Tier 1, 1. 소셜/이메일 로그인), "OAuth 2.0 표준 프로토콜, 소셜 플랫폼 API 무료 제공"이라고만 기술했다. **개인정보 수집·이용·처리에 대한 법적 요구사항은 전혀 언급되지 않았다.**

**반박 근거**: 소셜 로그인을 통해 수집되는 개인정보(이름, 이메일, 소셜 계정 식별자 등)는 「개인정보 보호법」의 적용을 받는다.

| 법적 요구사항 | 관련 법률 | 필요 조치 |
|--------------|----------|----------|
| 개인정보 수집·이용 동의 | 개인정보 보호법 제15조 | 동의 화면 구현, 동의 이력 관리 |
| 개인정보 처리방침 공개 | 개인정보 보호법 제30조 | 처리방침 문서 작성 및 웹사이트 공시 |
| 개인정보 보호책임자 지정 | 개인정보 보호법 제31조 | 책임자 지정 및 공개 |
| 개인정보 파기 | 개인정보 보호법 제21조 | 보유기간 경과 시 파기 절차 |
| 개인정보 안전성 확보 조치 | 개인정보 보호법 제29조, 시행령 제30조 | 접근 통제, 암호화, 접속 기록 관리 등 |
| 개인정보 처리 위탁 고지 | 개인정보 보호법 제26조 | 소셜 로그인 연동 시 위탁 관계 정리 |

**참고**: 서프컴퍼니도 개인정보 처리방침(9개 조항)과 개인정보보호책임자(한지성, jshan@surff.kr)를 운영 중이다(서프컴퍼니 사이트 분석 보고서 §1. 개요). 카고플랜이 자체 서비스를 구축할 경우, 동등 수준의 법적 준비가 필수이며 이에 대한 **법률 검토 비용, 문서 작성 비용, 시스템 구현 비용**이 원문 보고서에 반영되지 않았다.

### 4.2 소셜 로그인 심사/승인 절차

**원문 인용**: 원문 보고서는 소셜 로그인에 대해 "소셜 플랫폼 API 무료 제공"이라고만 기술했다(원문 §Tier 1, 1. 소셜/이메일 로그인, 데이터 확보 항목).

**반박 근거**: NAVER, KAKAO, GOOGLE 각 플랫폼의 로그인 API 사용에는 **사업자 심사, API 사용 승인, 보안 검수** 절차가 필요하다.

| 플랫폼 | 심사 절차 | 소요 기간 (추정) | 요구 사항 |
|--------|----------|-----------------|----------|
| NAVER 로그인 | 네이버 개발자 센터 앱 등록 → 검수 요청 → 승인 | 수일~수주 | 사업자 등록, 서비스 URL, 개인정보 처리방침 |
| KAKAO 로그인 | 카카오 디벨로퍼 앱 등록 → 비즈앱 전환 → 검수 | 수일~수주 | 사업자 등록, 개인정보 처리방침, 서비스 검수 |
| GOOGLE 로그인 | Google Cloud Console 프로젝트 → OAuth 동의 화면 검수 | 수일~수주 | 개인정보 처리방침, 사용 범위 심사 |

**이 심사 절차의 소요 기간과 준비 사항이 로드맵에 전혀 반영되지 않았다.** 특히 소셜 로그인이 "모든 개인화 기능의 전제조건"(Phase 1 선행 인프라)이라면, 이 심사 지연이 전체 로드맵에 미치는 영향도 분석되어야 한다.

### 4.3 정보통신망법 (뉴스레터)

**원문 인용**: 원문 보고서는 뉴스레터 구독을 Tier 2로 분류하면서(원문 §Tier 2, 11. 뉴스레터 구독), "이메일 구독 입력란(Footer 공통)"이라고만 기술했다. 이메일 마케팅에 적용되는 **법적 의무**는 언급되지 않았다.

**반박 근거**: 뉴스레터/이메일 마케팅은 「정보통신망 이용촉진 및 정보보호 등에 관한 법률」의 규제를 받는다.

| 법적 의무 | 관련 법률 | 위반 시 제재 |
|----------|----------|-------------|
| 수신자의 사전 동의 (opt-in) | 정보통신망법 제50조 제1항 | 3,000만원 이하 과태료 |
| 수신 거부 수단 제공 | 정보통신망법 제50조 제5항 | 3,000만원 이하 과태료 |
| 발송자 정보 표시 | 정보통신망법 제50조 제4항 | 과태료 |
| 수신 거부 처리 (거부 후 재발송 금지) | 정보통신망법 제50조 제7항 | 3,000만원 이하 과태료 |

원문 보고서의 "운영 인력: 콘텐츠 담당과 겸임" 산정에는 이러한 법적 의무 이행을 위한 **시스템 구현 비용(수신 동의 관리, 수신 거부 처리, 발송 기록 보관)과 법률 검토 비용**이 포함되어 있지 않다.

### 4.4 전자상거래법 (Booking)

**원문 인용**: 원문 보고서는 Booking 관리를 Tier 1으로 분류하면서(원문 §Tier 1, 4. Booking 관리 시스템), "FCL 정보 관리, 부킹 상태 추적, 고객별 관리"라고 기술했다.

**반박 근거**: Booking 관리가 온라인을 통한 거래 중개 또는 관리 기능을 포함한다면, 다음 법적 검토가 필요하다.

| 검토 항목 | 관련 법률 | 필요 조치 |
|----------|----------|----------|
| 통신판매업 신고 | 전자상거래 등에서의 소비자보호에 관한 법률 제12조 | 관할 지자체에 통신판매업 신고 |
| 이용약관 작성 | 약관의 규제에 관한 법률, 전자상거래법 | 서비스 이용약관 작성 및 공시 |
| 전자문서/전자서명 | 전자문서 및 전자거래 기본법 | 부킹 확인서 등 전자문서 효력 확보 |

**참고**: 서프컴퍼니는 통신판매업신고번호(2024-서울마포-2973)를 보유하고, 5장 25조의 서비스 이용약관을 운영 중이다(서프컴퍼니 사이트 분석 보고서 §1. 개요). 카고플랜이 유사한 온라인 부킹 기능을 구축할 경우, **동일한 법적 준비가 필요하나 원문 보고서에는 이 비용과 절차가 전혀 반영되지 않았다.**

### 4.5 공공데이터/선사 API 이용 제한

**원문 인용**: 원문 보고서는 다음과 같이 데이터 소스를 기술했다(원문 §핵심 데이터 소스 정리):

- 화물 추적: "선사 API (MSC, Maersk, ONE 등)" — "무료~유료", "대부분 공개 API 제공"
- 입출항 정보: "항만공사/해양수산부" — "공공데이터"
- 물동량: "관세청 수출입무역통계" — "공공데이터"

**반박 근거**:

#### 공공데이터 이용 제한

| 검토 항목 | 내용 |
|----------|------|
| 공공데이터 이용 약관 | 공공데이터의 제공 및 이용 활성화에 관한 법률에 따라, 공공데이터의 **상업적 이용 시 출처 표시 의무** 및 이용 약관 준수 필요 |
| API 호출 한도 | 공공데이터포털 API는 일별 호출 한도 존재. 서비스 규모에 따라 한도 초과 가능 |
| 데이터 품질 보장 | 공공데이터의 실시간성, 정확성은 제공 기관에 따라 상이하며, 서비스 품질 보장 불가 |

#### 선사 API 이용 제한

원문 보고서는 "공개 API 다수 존재"라고 기술했으나, **이 주장의 구체적 근거가 제시되지 않았다.**

| 검토 항목 | 내용 |
|----------|------|
| API 이용 약관 | 주요 선사(MSC, Maersk, ONE 등)의 API는 개별 이용 약관이 있으며, **상업적 사용에 대한 별도 계약**이 필요할 수 있음 |
| 호출량 제한 | 대부분의 선사 API는 **일별/월별 호출 한도**를 설정하며, 초과 시 유료 플랜 전환 필요 |
| 비즈니스 계약 요구 | 일부 선사 API는 개인 개발자가 아닌 **기업 단위 계약**을 요구 |
| API 중단/변경 위험 | 원문 보고서도 "유지보수 부담: 높음 — 선사별 API 스펙 변경 대응"을 인정했으나, 이것이 **구축 의사결정에 미치는 위험**은 충분히 반영되지 않음 |

### 4.6 이용약관/개인정보 처리방침 작성 비용

**원문 인용**: 원문 보고서 전체에서 이용약관 및 개인정보 처리방침 작성에 대한 언급은 없다.

**반박 근거**: 자체 서비스를 구축하여 회원가입, 부킹, 데이터 조회 등의 기능을 제공할 경우, 다음 법률 문서가 필수적이다:

| 필수 문서 | 참고 (서프컴퍼니) | 필요 비용 |
|----------|-----------------|----------|
| 서비스 이용약관 | 5장 25조 (서프 `/ko/termUse`) | 법률 사무소 검토/작성 비용 |
| 개인정보 처리방침 | 9개 조항 (서프 `/ko/termPersonal`) | 법률 사무소 검토/작성 비용 |

**서프컴퍼니 수준의 이용약관(5장 25조)과 개인정보 처리방침(9개 조항)을 법적으로 유효하게 작성하려면 전문 법률 검토가 필요하며, 이 비용이 자체 구축 비용에 포함되어야 한다.**

## 5. 논리적 결함에 대한 반박

### 5.1 자체 구축 긍정 편향

**원문 인용**: 원문 보고서의 분석 목적은 "서프컴퍼니가 제공하는 기능들을 분석하여, 카고플랜이 **서프와 계약하지 않고 직접 구축할 경우** 유효한 기능이 무엇인지 파악"(원문 §분석 목적)이다.

**반박 근거**: 이 목적 자체가 "자체 구축"을 전제하고 있다. **"구축하지 않는다"는 선택지를 진지하게 검토하지 않은 채**, 어떤 기능을 구축할 것인가만 분석한다. 이는 분석의 근본적 편향이다.

의사결정에 필요한 대안 분석 프레임워크:

| 대안 | 장점 | 단점 | 원문 검토 여부 |
|------|------|------|:------------:|
| **자체 구축** | 완전한 통제, 데이터 자사 보유 | 높은 비용, 긴 기간, 높은 리스크 | ✅ (편향적으로) |
| **SaaS 구매** (ShipGo 등) | 낮은 비용, 빠른 도입 | 커스터마이징 제한, 외부 의존 | ❌ (언급만) |
| **외주 개발** | 내부 인력 부족 해결 | 품질 관리, 유지보수 이관 | ❌ |
| **오픈소스 활용** | 비용 절감, 커뮤니티 지원 | 커스터마이징 필요, 보안 관리 | ❌ |
| **구축 안 함** | 핵심 사업 집중 | 디지털 역량 부재 | ❌ |

해상운임 시장 분석 보고서(§7.3)는 A~D 4가지 전략적 대안을 제시하고 "D. 단계적 접근"을 권고했다. 자체구축 보고서는 이 프레임워크를 무시하고 **자체 구축만을 결론으로 전제**했다.

### 5.2 고객 니즈 검증 없는 기능 확정

**원문 인용**: 원문 보고서는 서프컴퍼니의 16개 기능을 분석하여 5개를 "즉시 구축 추천"(Tier 1)으로 분류했다(원문 §Tier 1).

**반박 근거**: 해상운임 시장 분석 보고서(§7.1)는 단기 권고 1순위로 **"기존 고객 니즈 조사: 카고플랜 기존 고객의 디지털 서비스 실제 수요 파악"**을 제시했다.

자체구축 보고서는 **이 고객 니즈 조사가 수행되지 않은 상태에서** 서프컴퍼니의 기능 목록을 기준으로 구축 대상을 확정했다. 이는 다음과 같은 위험을 내포한다:

1. **서프의 고객과 카고플랜의 고객은 다르다**: 서프는 마켓플레이스(중소 화주 중심), 카고플랜은 포워더(대기업 중심). 서프에서 필요한 기능이 카고플랜 고객에게도 필요하다는 보장이 없다.
2. **공급자 중심 사고**: "만들 수 있는 것"이 아니라 "고객이 원하는 것"을 먼저 파악해야 한다.

### 5.3 ShipGo 브릿지 전략의 판단 기준 부재

**원문 인용**: 원문 보고서는 "ShipGo Plug-in vs 자체 구축 전략" 섹션(원문 §ShipGo Plug-in vs 자체 구축 전략)에서 다음 3단계를 제시했다:

| 단계 | 전략 | 기간 |
|------|------|------|
| 1단계 | ShipGo Plug-in 설치 | 즉시 (무료) |
| 2단계 | 카고 트래킹 자체 개발 | Phase 1 |
| 3단계 | ShipGo → 자체 시스템 마이그레이션 | Phase 1 완료 후 |

**반박 근거**: 이 전략에는 **전환 판단 기준(Go/No-Go Criteria)이 없다.**

| 미정의 항목 | 질문 |
|------------|------|
| 검증 기간 | ShipGo를 얼마나 사용한 후 전환을 판단하는가? |
| 성공 지표 | 어떤 KPI가 달성되면 자체 구축으로 전환하는가? (사용자 수? 조회 빈도? 고객 만족도?) |
| 실패 지표 | 어떤 조건에서 자체 구축을 포기하는가? |
| 비용 비교 시점 | ShipGo 유료 전환 비용 vs 자체 구축 TCO를 언제 비교하는가? |

전환 판단 기준 없이 "검증 후 전환"이라고만 서술하면, **사실상 자체 구축이 기정사실화**되는 결과를 낳는다.

### 5.4 "공개 API" 가정의 위험성

**원문 인용**: 카고 트래킹 데이터 확보에 대해 "선사 API 연동 필요 (공개 API 다수 존재)"(원문 §Tier 1, 2. 카고 트래킹, 데이터 확보 항목)라고 기술했다.

**반박 근거**: 이 주장에는 다음이 결여되어 있다:

1. **"공개 API 다수"의 구체적 목록**: MSC, Maersk, ONE, HMM, Evergreen 등 어떤 선사가 어떤 조건으로 API를 공개하는지 명시되지 않았다.
2. **"공개"의 정의**: 문서 공개(documentation public)와 무료 상업적 사용(free commercial use)은 다르다. API 문서가 공개되어 있더라도 상업적 사용에는 별도 계약이 필요할 수 있다.
3. **API 안정성**: 선사 API는 선사의 사업 판단에 따라 언제든 변경/중단될 수 있다. 원문 보고서도 유지보수 부담이 "높음"이라고 인정했으나, 이것이 **구축 추천 판정에는 반영되지 않았다**.

### 5.5 카고플랜 내부 역량 미확인

**원문 인용**: 원문 보고서는 "운영 역량 전제조건"(원문 §운영 역량 전제조건)에서 IT 운영 인력, 운영 담당자, 콘텐츠 담당자 등이 "반드시 갖추어져야 한다"고 기술했다.

**반박 근거**: 플랫폼 활용 분석 보고서(§4.2)도 "추가 확인 필요" 항목으로 **"현재 디지털 시스템 현황"**을 명시했다. 자체구축 보고서는 다음 내부 현황을 확인하지 않은 채 외부 분석만 수행했다:

| 미확인 항목 | 의사결정에 미치는 영향 |
|------------|---------------------|
| 현재 IT 인력 규모 및 역량 | Phase 1 착수 가능 여부 판단 불가 |
| 현재 IT 인프라 현황 | 추가 인프라 투자 규모 산정 불가 |
| 개발 예산 | 어느 Phase까지 실행 가능한지 판단 불가 |
| 경영진의 디지털 투자 의지 | 로드맵 실행력 판단 불가 |

**내부 역량이 파악되지 않은 상태에서 "Phase 1: 즉시 착수"라는 권고는 현실적 근거가 부족하다.**

## 6. 문서 구조/정합성 문제

### 6.1 "5단계 평가" 표기 오류

**원문 인용**: "자체 구축 유효성 분석 (5단계 평가)"(원문 §자체 구축 유효성 분석 (5단계 평가))

**반박 근거**: 실제 평가는 **Tier 1(즉시 구축 추천) / Tier 2(조건부 추천) / Tier 3(비추천)의 3단계**이다. "5단계 평가"가 무엇을 지칭하는지 정의가 없다. 평가 기준 테이블의 4개 항목(구축 난이도, 데이터 확보, 카고플랜 적합성, 종합 판정)을 5단계라고 하기에도 4개이다. 문서 제목과 내용이 불일치한다.

### 6.2 Tier-Phase 매핑 불일치

**원문 인용**: 기능별 Tier 분류와 Phase 배치를 교차 비교하면:

| 기능 | Tier | Phase | 불일치 |
|------|:----:|:-----:|:-----:|
| 해상 운임 조회 | 2 (조건부 추천) | 3 (선택적) | ⚠️ |
| 해운 데이터 맵 | 2 (조건부 추천) | 3 (선택적) | ⚠️ |
| 해운 대시보드 | 2 (조건부 추천) | 2 (Phase 1 완료 후) | ✅ |

**반박 근거**: Tier 2(조건부 추천)인 해상 운임 조회와 해운 데이터 맵이 Phase 3(선택적)에 배치되었다. 이는 Tier 분류의 기준(구축 유효성)과 Phase 배치의 기준(실행 우선순위)이 **다른 기준으로 작동한다**는 것을 의미하는데, 그 기준의 차이가 문서에 명시되어 있지 않다. 독자 입장에서 "조건부 추천인데 왜 선택적인가?"라는 혼동이 발생한다.

### 6.3 기능 분류 기준 혼재

**원문 인용**: 16개 기능 목록(원문 §서프컴퍼니 전체 기능 목록)을 보면:

| 유형 | 예시 |
|------|------|
| **페이지** | 마켓플레이스(`/marketplace`), 블로그(`/blog`) |
| **UI 컴포넌트** | Notification 시스템(사이드바), Booking 관리(사이드바) |
| **공통 요소** | 뉴스레터 구독(Footer 공통), 다국어 지원(Header) |
| **기능** | 소셜/이메일 로그인(`/login`) |
| **외부 서비스** | 해운 대시보드(`terminal.surff.kr`) |

**반박 근거**: 페이지, UI 컴포넌트, 공통 요소, 기능, 외부 서비스가 동일 레벨에서 비교되고 있다. 이는 구축 규모와 복잡도를 비교할 때 혼란을 초래한다. 예를 들어 "뉴스레터 구독"(Footer의 이메일 입력란)과 "해상 운임 조회"(전체 페이지 + 데이터 수집)를 동일한 "기능"으로 비교하는 것은 적절하지 않다.

### 6.4 FTE 산정 비현실성

**원문 인용**: 운영 종합 매트릭스(원문 §기능별 운영 부담 요약)에서 기능별 FTE가 개별 제시되었다.

**반박 근거**:

1. **Phase별 합산 FTE와 중복 미고려**: Phase 1의 FTE를 단순 합산하면 3.0명이지만, 한 사람이 여러 기능을 담당할 수 있다. 실제 필요 인원은 단순 합산보다 적을 수 있으나, 그 조정이 이루어지지 않았다.
2. **0.1명, 0.2명 단위의 비현실성**: 운임지수 운영에 "IT 담당 0.1명"이라는 것은 한 사람이 10개 기능을 동시에 담당한다는 의미인데, 이러한 멀티태스킹의 현실성이 검토되지 않았다.
3. **개발 인력 누락**: 위 FTE는 모두 "운영 인력"이다. 구축 단계의 개발 인력(설계, 개발, 테스트, 배포)은 별도로 산정되어야 하나 전혀 제시되지 않았다.

### 6.5 ASCII 매트릭스 배치 일관성

**원문 인용**: "구축 난이도 vs 운영 부담" ASCII 차트(원문 §구축 난이도 vs 운영 부담 매트릭스)에서 "소셜/이메일 로그인"이 "운영 부담 중간" 행에 배치되어 있다.

**반박 근거**: 기능별 운영 부담 요약 테이블에서 소셜/이메일 로그인의 구축 난이도는 **"낮음~중간"**으로 기술되어 있으나, 유지보수 부담은 "중간", CS 부담은 "중간"이다. ASCII 차트에서 "운영 부담 중간"에 배치한 기준이 명확하지 않다. **운영 부담의 정의(어떤 항목들의 가중 평균인지)가 명시되지 않아** 차트 배치의 객관성을 검증할 수 없다.

## 7. 타 문서 정합성 문제

### 7.1 시장 보고서 권고 사항 미이행

**원문 인용**: 해상운임 시장 분석 보고서(§7.1)의 단기 권고(1-3개월):

| 우선순위 | 항목 | 자체구축 보고서 반영 여부 |
|----------|------|:---------------------:|
| 1 | 기존 고객 니즈 조사 | ❌ 미이행 |
| 2 | 비용-편익 분석 | ❌ 미이행 |
| 3 | 경쟁사 실제 성과 검증 | ❌ 미이행 |

**반박 근거**: 시장 보고서는 **"기존 고객 니즈 조사 → 비용-편익 분석 → 파일럿 프로젝트"** 순서를 명확히 권고했다. 자체구축 보고서는 이 세 단계를 **모두 건너뛰고 곧바로 기능 분류와 로드맵 확정**으로 진행했다. 이는 시장 보고서의 핵심 결론인 "검증된 실패 사례와 구조적 한계를 고려한 신중한 접근"(시장 보고서 §Executive Summary)과 정면으로 배치된다.

### 7.2 활용 보고서 전략과의 충돌

**원문 인용**: 플랫폼 활용 분석 보고서(§Executive Summary)의 핵심 결론은 **"플랫폼 입점과 솔루션 도구 활용을 병행하는 것이 최적 전략"**이다.

**반박 근거**: 자체구축 보고서는 "기존 플랫폼 활용과 자체 구축은 상호 배타적이 아닌 단계적 전략"(원문 §종합 권고)이라고 정리했으나, 실제 내용을 보면 **최종적으로 자체 구축 전환을 전제**하고 있다:

- ShipGo Plug-in → "자체 구축으로 전환하면 데이터 자사 보유 + 커스터마이징이라는 추가 이점 확보 가능" (원문 §Tier 1, 2. 카고 트래킹 근거)
- 3단계 전략의 최종 목표: "외부 의존도 제거, 완전 자사 운영" (원문 §ShipGo Plug-in vs 자체 구축 전략)

활용 보고서의 "병행" 전략과 자체구축 보고서의 "단계적 전환(최종 자체 구축)" 전략은 **근본적으로 다른 방향**이다. 활용 보고서는 외부 플랫폼의 지속적 활용을 전제하고, 자체구축 보고서는 외부 의존도 제거를 최종 목표로 한다.

### 7.3 비용-편익 프레임워크 미적용

**원문 인용**: 해상운임 시장 분석 보고서(§5.1)가 제시한 비용-편익 분석 프레임워크:

| 항목 | 시장 보고서 상태 | 자체구축 보고서 반영 |
|------|:---------------:|:------------------:|
| 초기 개발 비용 | "추가 조사 필요" | ❌ 여전히 미해결 |
| 운영 비용 | "추가 조사 필요" | ❌ 여전히 미해결 |
| 손익분기점 | "추가 조사 필요" | ❌ 여전히 미해결 |
| ROI 기간 | "추가 조사 필요" | ❌ 여전히 미해결 |

**반박 근거**: 시장 보고서가 v2.0에서 의도적으로 추가한 비용-편익 분석 프레임워크의 **4개 항목 전부**가, 자체구축 보고서에서도 채워지지 않았다. 이 프레임워크가 "추가 조사 필요"인 상태에서 "Phase 1: 즉시 착수"를 권고하는 것은 **비용-편익 분석 없는 투자 의사결정**이다.

### 7.4 서프 벤치마킹의 구조적 한계

**원문 인용**: 원문 보고서는 서프컴퍼니의 16개 기능을 분석 기준으로 사용했다(원문 §서프컴퍼니 전체 기능 목록).

**반박 근거**: 해상운임 시장 분석 보고서(§3.3)는 서프컴퍼니 벤치마킹 시 다음 주의점을 명시했다:

| 고려사항 | 시장 보고서 경고 | 자체구축 보고서 반영 |
|----------|----------------|:------------------:|
| 대기업 지원 | "현대글로비스 사내벤처로 초기 자금/인프라 지원 가능" | ❌ 미반영 |
| 물량 확보 | "현대그룹 네트워크 활용 가능성" | ❌ 미반영 |
| 실제 거래량 미공개 | "벤치마킹 시 실제 성과 검증 필요" | ❌ 미반영 |
| 수익 모델 미검증 | "플랫폼 중개 수수료의 수익성 확인 어려움" | ❌ 미반영 |

서프컴퍼니는 현대글로비스 사내벤처로서 **초기 자금, 인프라, 물량 확보에서 구조적으로 유리한 환경**에서 성장했다. 카고플랜이 동일한 기능을 구축한다고 해서 동일한 효과를 얻을 수 있다는 보장이 없으며, 이러한 구조적 차이가 자체구축 보고서에 반영되지 않았다.

## 9. 결론

원문 보고서 「카고플랜 자체 구축 가능 기능 정밀 분석 v2.0」은 서프컴퍼니 16개 기능에 대한 체계적 분류와 운영 요소 분석이라는 점에서 가치가 있다. 그러나 **투자 의사결정의 핵심 요소인 비용 분석과 법적 검토가 전면 부재**하며, **자체 구축을 전제한 논리적 편향**이 있고, **선행 보고서들의 권고를 이행하지 않은 채 로드맵을 확정**했다는 점에서 중대한 결함이 있다.

### 핵심 메시지

> **"무엇을 만들 수 있는가"보다 "만드는 것이 사는 것보다 나은가", "우리가 만들 역량이 있는가", "법적으로 준비가 되어 있는가"를 먼저 답해야 한다.**

이 보고서를 기반으로 자체 구축 의사결정을 내리기 전에, §8의 권고 사항 — 특히 **고객 니즈 조사, 비용-편익 분석, 법적 검토, 내부 역량 파악** — 이 선행되어야 한다.
