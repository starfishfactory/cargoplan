# 카고플랜 자체 구축 가능 기능 정밀 분석

**프로젝트**: 카고플랜 자체 구축 기능 검토
**작성일**: 2026-02-05
**버전**: 3.0 (반박 검토서 12개 권고 반영)

---

## 1. 분석 목적

서프컴퍼니(surff.kr)가 제공하는 기능들을 분석하여, 카고플랜이 **서프와 계약하지 않고 직접 구축할 경우** 유효한 기능이 무엇인지 파악한다. v2.0에서는 **구축 후 실제 운영 가능성**까지 포함한 종합 판단을 제공하며, v3.0에서는 반박 검토서의 12개 권고를 반영하여 **Build vs Buy 의사결정에 필요한 비용·법적·역량 분석**을 추가한다.

### 교차 검증 문서

| 문서 | 반영 내용 |
|------|----------|
| 서프컴퍼니 사이트 분석 보고서 | 페이지별 상세 기능 교차 확인 |
| 카고플랜 플랫폼 활용 분석 보고서 | ShipGo Plug-in vs 자체 구축 전략 정합성 |
| 해상운임 시장 분석 보고서 | 디지털 포워딩 실패 사례, 시장 리스크, 비용-편익 프레임워크(§5.1), 권고 순서(§7.1) |
| 카고플랜 자체구축 기능 분석 보고서 반박 검토서 | 12개 권고사항 전수 반영 (비용/법적/논리/구조/정합성) |
| surff-html-sitemap / visual-sitemap | 16개 페이지 URL 및 기능 구조 정합성 |

## 2. 의사결정 전제조건

> **반박 검토서 권고 #1, #4 반영**: 기능 선정 전 고객 니즈와 내부 역량 확인이 선행되어야 한다.

### 2.1 고객 니즈 조사 프레임워크

해상운임 시장 분석 보고서(§7.1)는 단기 권고 1순위로 "기존 고객 니즈 조사"를 제시했다. 서프컴퍼니의 기능 목록이 아닌 **카고플랜 고객의 실제 수요**를 기반으로 기능을 선정해야 한다.

| 조사 항목 | 대상 | 방법 | 상태 |
|----------|------|------|:----:|
| ⬜ 대기업 고객 디지털 서비스 수요 | KT&G, 명보 등 기존 고객 | 직접 인터뷰 (3~5개사) | 추후 보완 |
| ⬜ 화물 추적 서비스 사용 빈도/만족도 | 기존 고객 + 잠재 고객 | 설문조사 (20개사+) | 추후 보완 |
| ⬜ 온라인 부킹 전환 의향 | 기존 고객 | 직접 인터뷰 | 추후 보완 |
| ⬜ 경쟁 포워더 디지털 서비스 비교 | 시장 전체 | 벤치마킹 조사 | 추후 보완 |
| ⬜ 위험물/프로젝트 화물 디지털화 니즈 | 특수 화물 고객 | 직접 인터뷰 | 추후 보완 |

**참고**: 서프의 고객(마켓플레이스 중소 화주)과 카고플랜의 고객(대기업 중심 포워딩)은 세그먼트가 다르다. 서프에서 필요한 기능이 카고플랜 고객에게도 필요하다는 보장이 없으며, 이 조사가 완료되기 전까지 본 보고서의 기능 선정은 **잠정적 판단**임을 유의해야 한다.

### 2.2 내부 역량 현황 파악 프레임워크

플랫폼 활용 분석 보고서(§4.2)도 "추가 확인 필요" 항목으로 "현재 디지털 시스템 현황"을 명시했다.

| 파악 항목 | 필요 사유 | 상태 |
|----------|----------|:----:|
| ⬜ 현재 IT 인력 규모 및 역량 | Phase 1 착수 가능 여부 판단 | 추후 보완 |
| ⬜ 현재 IT 인프라 현황 | 추가 인프라 투자 규모 산정 | 추후 보완 |
| ⬜ 연간 IT 예산 | 어느 Phase까지 실행 가능한지 판단 | 추후 보완 |
| ⬜ 경영진 디지털 투자 의지 | 로드맵 실행력 판단 | 추후 보완 |
| ⬜ 기존 시스템 (ERP, 메일 등) 현황 | 통합 연동 범위 결정 | 추후 보완 |

> **⚠️ 주의**: 위 항목이 파악되지 않은 상태에서 "Phase 1: 즉시 착수"라는 권고는 현실적 근거가 부족하다. §9 로드맵은 이 전제조건 확인 후 Phase 0(검증 단계)부터 시작하는 것으로 수정되었다.

## 3. 법적/규제 요구사항

> **반박 검토서 권고 #3, #7 반영**: 자체 서비스 구축 시 필수 법적 요구사항을 사전에 검토해야 한다.

### 3.1 개인정보 보호법

소셜/이메일 로그인을 통해 수집되는 개인정보(이름, 이메일, 소셜 계정 식별자 등)는 「개인정보 보호법」의 적용을 받는다.

| 조항 | 내용 | 필요 조치 | 비용 영향 |
|------|------|----------|----------|
| **제15조** (수집·이용) | 개인정보 수집·이용 시 정보주체 동의 필요 | 동의 화면 구현, 동의 이력 관리 시스템 | 개발 비용 포함 |
| **제21조** (파기) | 보유기간 경과·목적 달성 시 지체 없이 파기 | 자동 파기 배치 프로세스 구현 | 개발 비용 포함 |
| **제26조** (위탁) | 개인정보 처리 위탁 시 고지 의무 | 소셜 로그인 연동 위탁 관계 정리, 위탁 고지 | 법률 검토 비용 |
| **제29조** (안전조치) | 접근 통제, 암호화, 접속 기록 관리 등 | 보안 인프라 구축 (SSL, DB 암호화, 접근 로그) | 인프라 비용 포함 |
| **제30조** (처리방침) | 개인정보 처리방침 수립·공개 | 처리방침 문서 작성 및 웹사이트 공시 | 법률 사무소 200~400만원 |
| **제31조** (보호책임자) | 개인정보 보호책임자 지정·공개 | 책임자 지정 및 공시 | 내부 인사 조치 |

**참고**: 서프컴퍼니도 개인정보 처리방침(9개 조항)과 보호책임자(한지성, jshan@surff.kr)를 운영 중이다. 카고플랜 자체 구축 시 동등 수준의 법적 준비가 필수이다.

### 3.2 정보통신망법 — 뉴스레터

뉴스레터/이메일 마케팅은 「정보통신망 이용촉진 및 정보보호 등에 관한 법률」의 규제를 받는다.

| 조항 | 의무 | 위반 시 제재 |
|------|------|-------------|
| **제50조 제1항** | 수신자의 사전 동의 (opt-in) 필수 | 3,000만원 이하 과태료 |
| **제50조 제4항** | 발송자 정보 및 수신 거부 방법 표시 의무 | 과태료 |
| **제50조 제5항** | 수신 거부를 회피·방해하는 기술적 조치 금지 | 3,000만원 이하 과태료 |
| **제50조 제7항** | 수신 동의/거부/철회 처리 결과 통지 의무 | 3,000만원 이하 과태료 |

**시스템 구현 필요**: 수신 동의 관리, 수신 거부 처리, 발송 기록 보관 기능이 뉴스레터 시스템에 포함되어야 한다.

Booking 관리가 온라인을 통한 거래 관리 기능을 포함할 경우 다음이 필요하다.

| 조항 | 내용 | 필요 조치 |
|------|------|----------|
| **제12조** (통신판매업 신고) | 온라인 거래 시 관할 지자체에 통신판매업 신고 | 신고 절차 수행 |

**참고**: 서프컴퍼니는 통신판매업신고번호(2024-서울마포-2973)를 보유하고 있다.

### 3.4 공공데이터/선사 API 이용 제한

| 데이터 소스 | 이용 시 고려사항 |
|------------|----------------|
| **공공데이터** (관세청, 항만공사) | 공공데이터법에 따른 출처 표시 의무, 일별 API 호출 한도 존재, 실시간성·정확성 보장 불가 |
| **선사 API** (MSC, Maersk, ONE 등) | 상업적 사용 별도 계약 가능, 일별/월별 호출 한도, 선사 판단에 따른 변경/중단 위험 |

> **⚠️ 주의**: v2.0에서 "공개 API 다수 존재"로 기술했으나, API 문서 공개(documentation public)와 무료 상업적 사용(free commercial use)은 다르다. 각 선사별 API 이용 약관과 상업적 사용 조건을 사전에 확인해야 한다.

소셜 로그인 API 사용에는 사업자 심사, API 사용 승인, 보안 검수 절차가 필요하다.

| 플랫폼 | 심사 절차 | 소요 기간 (추정) | 사전 요구사항 |

| **NAVER** | 네이버 개발자 센터 앱 등록 → 검수 요청 → 승인 | 수일~수주 | 사업자 등록, 서비스 URL, 개인정보 처리방침 |

## 5. 자체 구축 유효성 분석 (3-Tier 평가)

> **v3.0 수정**: v2.0의 "5단계 평가" 표기를 실제 평가 체계인 "3-Tier 평가"로 정정. Tier-Phase 매핑 기준을 명시.

### Tier-Phase 매핑 기준

Tier(구축 유효성)와 Phase(실행 우선순위)는 **다른 기준**으로 작동한다.

| 기준 | 의미 | 판단 요소 |
|------|------|----------|
| **Tier** | 자체 구축의 유효성 | 기술 난이도, 데이터 확보 가능성, 비즈니스 적합성 |
| **Phase** | 실행 우선순위 | 고객 가치, 선행 조건 충족 여부, 운영 역량, 비용 대비 효과 |

따라서 Tier 2(조건부 추천)인 기능이 Phase 3(선택적)에 배치될 수 있다. 예: 해상 운임 조회는 구축 자체는 "조건부 가능"이나, 데이터 확보와 운영 부담이 높아 실행 우선순위는 낮다.

### 평가 기준

| 항목 | 설명 |
|------|------|
| **구축 난이도** | 기술적 복잡도 (낮음/중간/높음) |
| **데이터 확보** | 필요 데이터의 자체 확보 가능성 |
| **카고플랜 적합성** | 포워더 비즈니스에 실질적 도움 여부 |
| **종합 판정** | ✅ 구축 추천 / ⚠️ 조건부 추천 / ❌ 비추천 |

#### 운영 요소 평가 (v2.0 추가)

| 운영 요소 | 설명 |
|-----------|------|
| **어드민 구성** | 필요한 관리자 화면/도구 (설정, 모니터링, 데이터 관리 등) |
| **운영 인력** | 일상 운영에 필요한 인원/역할 |
| **관리자 교육** | 운영 담당자 숙련에 필요한 교육 수준 (낮음/중간/높음) |

| **고객 지원 부담** | 기능 도입 시 예상되는 CS 이슈 |

### ✅ Tier 1: 즉시 구축 추천 (높은 실효성)

#### 1. 소셜/이메일 로그인 (필수 인프라)
| 항목 | 평가 |
|------|------|
| 구축 난이도 | **낮음~중간** |
| 데이터 확보 | OAuth 2.0 표준 프로토콜, 소셜 플랫폼 API 무료 제공 |
| 카고플랜 적합성 | **필수** — 모든 개인화 기능(트래킹, 알림, 부킹)의 전제조건 |
| 핵심 기능 | 이메일 인증(입력→인증코드→간편 회원가입), NAVER/KAKAO/GOOGLE 소셜 로그인 |
| 자체 구축 이점 | 고객 데이터 자사 보유, 향후 CRM 연동 기반 |
| **어드민 구성** | 회원 관리 대시보드(가입/탈퇴/정지), 권한 관리, 로그인 이력 모니터링 |
| **운영 인력** | IT 담당 0.2명 (초기 설정 후 자동 운영) |
| **관리자 교육** | **낮음** — 회원 관리 UI 기본 교육 |
| **콘텐츠 관리** | 없음 (시스템 자동 운영) |
| **유지보수 부담** | **중간** — 소셜 로그인 API 변경 대응(NAVER/KAKAO/GOOGLE 각각), 보안 패치 |
| **고객 지원 부담** | **중간** — 로그인 오류, 비밀번호 분실, 인증 메일 미수신 CS 발생 |

**근거**: 카고 트래킹, 부킹 관리, 알림 등 모든 개인화 기능의 전제조건. Phase 1의 첫 번째 선행 인프라로 반드시 필요.

#### 2. 카고 트래킹 (BL 기반 화물 추적)
| 항목 | 평가 |
|------|------|
| 구축 난이도 | **중간** |
| 데이터 확보 | 선사 API 연동 필요 (공개 API 다수 존재) |
| 카고플랜 적합성 | **매우 높음** — 기존 고객(KT&G, 명보 등) 서비스 강화 핵심 |
| 핵심 기능 | BL 등록 폼(BL번호/선사/담당자), 일괄 등록, 엑셀 양식 다운로드/내보내기, 트래킹 테이블(BL/선사/Vessel/컨테이너/POL/POD/ETD/ETA/상태/담당자) |
| 자체 구축 이점 | 고객 데이터 100% 자사 보유, 커스터마이징 자유 |
| **어드민 구성** | 선사 API 연동 상태 모니터링, BL 일괄 관리, 고객별 트래킹 현황 대시보드, API 호출량/에러 모니터링 |
| **운영 인력** | IT 담당 0.5명 (API 모니터링) + 운영 담당 0.3명 (고객 문의 대응) |
| **관리자 교육** | **중간** — 선사 API 구조 이해, 트래킹 상태 해석, 장애 대응 |
| **콘텐츠 관리** | 없음 (실시간 API 데이터, 자동 갱신) |
| **유지보수 부담** | **높음** — 선사별 API 스펙 변경 대응(MSC, Maersk, ONE 등 각각 상이), 신규 선사 추가, 데이터 정합성 검증 |
| **고객 지원 부담** | **중간~높음** — BL 조회 실패, ETD/ETA 불일치, 선사 미지원 문의 |

**근거**: 트레드링스 ShipGo Plug-in이 무료로 제공할 정도로 핵심 니즈. 포워더가 직접 운영하면 고객 신뢰도 상승.

> **플랫폼 활용 보고서와의 관계**: 활용 보고서는 "ShipGo Plug-in 즉시 도입"을 권고하고 있으며, 이는 자체 구축 전 **단기 브릿지 전략**으로 유효함. ShipGo Plug-in으로 고객 니즈를 검증한 뒤, 자체 구축으로 전환하면 데이터 자사 보유 + 커스터마이징이라는 추가 이점 확보 가능.

> **FTE 해석 주의** (v3.0 추가):
> - 위 FTE는 **운영 인력**만 산정한 것이다. **개발(구축) 인력은 별도**이며 §7 비용-편익 분석에서 산정한다.

### 구축 난이도 vs 운영 부담 매트릭스

> **배치 기준** (v3.0 추가): X축(구축 난이도)은 기능별 "구축 난이도" 항목 기준. Y축(운영 부담)은 유지보수 부담, CS 부담, 콘텐츠 부담 3개 항목의 **종합 판단** 기준이다. (높음이 2개 이상 → 고, 중간 이상 2개 이상 → 중, 나머지 → 저)

```
운영 부담 높음 │  해상 운임 조회    해운 데이터 맵
               │  Booking 관리
               │  블로그(콘텐츠)
               │
운영 부담 중간 │  카고 트래킹       해운 대시보드
               │  선박 입출항 알림
               │  소셜/이메일 로그인
               │
운영 부담 낮음 │  실시간 알림       운임지수
               │  뉴스레터          수출입 물동량
               │
               └──────────────────────────────────
                 구축 난이도 낮음    구축 난이도 중간    구축 난이도 높음
```

**해석**:
- **우측 상단 (고난이도 + 고운영)**: 해상 운임 조회, 해운 데이터 맵 → 가장 신중한 접근 필요
- **좌측 하단 (저난이도 + 저운영)**: 실시간 알림, 뉴스레터, 운임지수 → 빠른 도입 가능
- **Booking 관리**: 구축은 중간이나 운영 부담 높음 → 운영 역량 확보가 관건
- **카고 트래킹**: 구축 중간, 운영 중간이나 **유지보수 부담 높음** → 선사 API 관리 전담 필요

### 7.6 ROI 분석 및 손익분기점

> **⬜ 추후 정밀 산정 필요**: ROI 산정을 위해서는 자체 구축으로 인한 **수익 증가 또는 비용 절감 효과**를 정량화해야 한다. 현재 시점에서는 다음 항목이 확인되지 않아 정밀 산정이 불가하다.

| 미확인 항목 | 필요 사유 |
|------------|----------|
| ⬜ 자체 구축으로 인한 고객 이탈 방지 효과 | 리텐션 기여 매출 산정 |
| ⬜ 디지털 서비스로 인한 신규 고객 획득 효과 | 신규 매출 산정 |
| ⬜ 운영 효율화로 인한 인건비 절감 효과 | 비용 절감 산정 |
| ⬜ 외부 솔루션(ShipGo 등) 유료 전환 시 비용 | Buy 대안 비용 산정 |

**예비 판단**: 5년 TCO 약 8~15.7억원을 회수하려면, 자체 구축이 연간 **1.6~3.1억원 이상**의 추가 매출 또는 비용 절감을 창출해야 한다. 이 수준이 현실적인지는 §2의 고객 니즈 조사 및 내부 역량 파악 후 판단할 수 있다.

## 8. Build vs Buy 비교 분석

> **반박 검토서 권고 #5 반영**: "자체 구축"만을 전제한 분석 편향을 보완하기 위해 5가지 대안을 비교한다.

### 8.1 5가지 대안 비교표

| 항목 | 자체 구축 | SaaS 구매 (ShipGo 등) | 외주 개발 | 오픈소스 활용 | 미구축 |
|------|----------|---------------------|----------|-------------|--------|
| **초기 비용** | 높음 (2~4억) | 낮음 (무료~월 수십만) | 중간~높음 | 낮음~중간 | 없음 |
| **운영 비용** | 높음 (연 0.9~1.7억) | 낮음 (월 구독료) | 중간 | 중간 | 없음 |
| **구축 기간** | 12~18개월 | 즉시~수주 | 6~12개월 | 3~6개월 | - |
| **커스터마이징** | 완전 자유 | 제한적 | 가능 (계약 범위) | 가능 (기술력 필요) | - |
| **데이터 소유** | 100% 자사 | 외부 보관 | 자사 (계약에 따라) | 자사 | - |
| **유지보수** | 자사 부담 | 서비스 제공자 | 이관 리스크 | 자사 부담 | - |
| **기술 리스크** | 내부 역량 의존 | 서비스 중단 위험 | 품질 관리 어려움 | 커뮤니티 의존 | 디지털 역량 부재 |
| **핵심 장점** | 완전한 통제 | 빠른 도입, 낮은 비용 | 내부 인력 부족 해결 | 비용 절감 | 핵심 사업 집중 |
| **핵심 단점** | 높은 비용·리스크 | 외부 의존, 차별화 제한 | 유지보수 이관 문제 | 보안·안정성 관리 | 시장 도태 위험 |

### 8.2 기능별 최적 조달 방식

| 기능 | 자체 구축 | SaaS | 외주 | OSS | 권장 |
|------|:--------:|:----:|:----:|:---:|------|

| 고객 만족도 | 분기별 설문 (NPS) | NPS 30+ |
| 조회 응답 시간 | 서비스 모니터링 | 3초 이내 |
| 고객별 조회 빈도 | 로그 분석 | 주 2회+ 사용 고객 50%+ |

### 10.3 판단 시점

| 시점 | 판단 내용 |
|------|----------|
| **ShipGo 도입 3개월 후** | 고객 사용률 및 피드백 1차 리뷰 |
| **ShipGo 도입 6개월 후** | Go/No-Go 판단 (위 기준 적용) |
| **ShipGo 유료 전환 시점** | 비용 비교 기반 최종 판단 |

> **No-Go 시**: ShipGo를 계속 활용하며 비용 효율성을 유지한다. "외부 의존도 제거"가 반드시 최선은 아니며, **활용 보고서의 "병행" 전략**이 더 합리적일 수 있다.

## 11. 리스크 분석

### 디지털 포워딩 시장 리스크

해상운임 시장 분석 보고서에서 확인된 핵심 리스크를 자체 구축 관점에서 재평가한다.

#### 글로벌 실패 사례 시사점

> "플랫폼에서 자동견적을 받고, 서류를 화주가 업로드하고, 서류 다운로드, 인보이스확인 등 건별로 가시성을 뚫어준건 맞으나, **기존에 이메일로 서류 몇번 주고받고 했을 때랑 크게 화주 입장에서 편해진점이 없다**" — 포워더넷 커뮤니티

| 시사점 | 자체 구축 반영 |
|--------|--------------|
| 화주 체감 편의가 핵심 | Tier 1 기능은 **실무 편의성**(화물 위치 실시간 확인, 부킹 상태 자동 알림)에 집중 → 플랫폼 실패 사례와 차별화 |
| 기술만으로 부족 | 기존 고객 관계(KT&G, 명보 등)를 디지털 서비스로 **강화**하는 접근, 신규 플랫폼 사업 진출과는 다름 |
| 수익성 증명 필요 | Tier 1은 **비용 절감**(운영 효율화) + **고객 리텐션** 중심 → 직접 수익 창출 대신 핵심 사업 강화 |

#### 플랫폼 수익성 한계

> "플랫폼은 어디까지나 중개자 역할이기 때문에 규모의 경제를 만들기에는 어려움이 있다" — 코리아로지스틱스뉴스

| 리스크 | 카고플랜 대응 |
|--------|-------------|
| 중개 수수료 모델의 한계 | 카고플랜은 **포워더**이므로 중개가 아닌 자사 서비스 도구로 구축 → 수익 모델 이슈 해당 없음 |
| 규모의 경제 달성 어려움 | Tier 3(마켓플레이스, 데이터 판매) 비추천의 근거. Tier 1~2는 자사 운영 도구이므로 규모 불필요 |

### ShipGo Plug-in vs 자체 구축 전략

플랫폼 활용 분석 보고서는 "ShipGo Plug-in 즉시 도입"을, 자체구축 보고서는 "카고 트래킹 자체 구축"을 권고한다. 이 두 접근은 **상충이 아닌 단계적 전략**으로 정리되며, 전환 판단 기준은 §10에서 별도 정의한다.

| 단계 | 전략 | 기간 | 목적 |
|------|------|------|------|
| **1단계: 즉시 도입** | ShipGo Plug-in 설치 | 즉시 (무료) | 고객 니즈 검증, 즉각적 서비스 개선 |
| **2단계: 자체 구축** | 카고 트래킹 자체 개발 | Phase 1 | 고객 데이터 자사 보유, 커스터마이징, Plug-in 의존도 탈피 |
| **3단계: 전환** | ShipGo → 자체 시스템 마이그레이션 | Phase 1 완료 후 | 외부 의존도 제거, 완전 자사 운영 |

### 운영 관점 리스크

| 리스크 | 심각도 | 설명 | 대응 방안 |
|--------|:------:|------|----------|
| 운영 인력 부족 | **높음** | Phase 1만 해도 IT 1.5명 + 운영 1.5명 수준 필요 | 단계적 도입으로 점진적 인력 확보 |
| 선사 API 의존도 | **높음** | 선사 API 변경/중단 시 핵심 서비스 장애 | 다중 선사 지원, 장애 대응 프로세스 수립 |
| 콘텐츠 생산 역량 | **중간** | 블로그/뉴스레터 지속 발행 부담 | Phase 2 이후 도입, 외부 협력 검토 |
| 기술 부채 축적 | **중간** | 다수 외부 API 연동으로 유지보수 복잡도 증가 | API 통합 레이어 설계, 모니터링 자동화 |
| 고객 기대치 관리 | **중간** | 디지털 서비스 도입 후 품질 기대 상승 | 점진적 기능 공개, 베타 테스트 운영 |

## 14. 최종 결론

### 의사결정 순서

> **반박 검토서 권고 #10 반영**: 시장 보고서 권고 순서를 준수한다.

자체 구축 의사결정은 다음 순서로 진행되어야 한다:

1. **고객 니즈 조사** (§2.1) — "만들 수 있는 것"이 아니라 "고객이 원하는 것" 확인
2. **내부 역량 파악** (§2.2) — "만들 역량이 있는가" 확인
3. **법적 준비** (§3) — "법적으로 준비가 되어 있는가" 확인
4. **비용-편익 분석** (§7) — "만드는 것이 사는 것보다 나은가" 확인
5. **Build vs Buy 판단** (§8) — 기능별 최적 조달 방식 결정
6. **파일럿 실행** (§9 Phase 0) — 소규모 검증
7. **단계적 확대** (§9 Phase 1~3) — Go/No-Go 게이트 기반

### 자체 구축 유효 기능 5가지

위 의사결정 과정을 거친 후, 잠정적으로 자체 구축에 **가장 유효한 기능 5가지**:

1. **소셜/이메일 로그인** — 모든 개인화 기능의 필수 전제조건 (Phase 1 선행 인프라)
2. **카고 트래킹** — 고객 가치 최고, 선사 API 활용으로 구현 가능
3. **Booking 관리 시스템** — 내부 운영 + 고객 서비스 동시 해결
4. **실시간 알림** — 위 두 기능의 가치를 극대화하는 보조 기능
5. **선박 입출항 알림** — 공공데이터 기반, 실무 즉시 활용

### 운영 역량 전제조건

자체 구축의 성공을 위해 다음 운영 역량이 반드시 갖추어져야 한다:

| 전제조건 | 상세 | Phase |
|----------|------|:-----:|
| **IT 운영 인력** | API 연동/모니터링 전담 최소 1명 | 1 |
| **운영 담당자** | 부킹 관리, 고객 대응 최소 1명 | 1 |
| **장애 대응 체계** | 선사 API 장애 시 수동 대응 프로세스 | 1 |
| **콘텐츠 담당자** | 블로그/뉴스레터 전담 또는 겸임 | 2 |
| **데이터 품질 관리** | 운임/물동량 데이터 정합성 검증 | 2~3 |

### 비용 요약

| 항목 | Phase 1 | Phase 1+2 | 비고 |
|------|:-------:|:---------:|------|
| 개발비 (일회성) | 0.9~1.8억 | 1.5~2.9억 | §7.1 참조 |
| 개발 인건비 (일회성) | 1.2~2.2억 | 1.8~3.3억 | §7.3 참조 |
| 인프라 (연간) | 636~1,740만 | 1,116~2,760만 | §7.2 참조 |
| 운영 인건비 (연간) | 8,500~16,500만 | 8,500~16,500만 | §7.3 참조 |
| **5년 TCO** | | **8~15.7억** | §7.4 참조 |

### 종합 권고

반면, **마켓플레이스/운임비교/데이터 판매**는 플랫폼 사업자(서프, 트레드링스)의 영역으로, 포워더인 카고플랜이 직접 구축하는 것은 비효율적이고 비즈니스 모델과 상충됨.

**기존 플랫폼 활용(ShipGo Plug-in 등)과 자체 구축은 상호 배타적이 아닌 단계적 전략**으로 접근해야 한다. 즉시 도입 가능한 외부 솔루션으로 고객 니즈를 검증한 뒤(Phase 0), Go/No-Go 게이트를 통과한 경우에만 검증된 영역부터 자체 구축으로 전환하는 것이 리스크를 최소화하는 최적 경로이다. No-Go 판단 시에는 **SaaS 활용 유지**가 합리적 대안이다 (§10, §13 참조).

> **핵심 메시지**: "남의 기능을 전부 따라 만들 필요 없다. 포워더로서 고객에게 직접 가치를 주는 기능에 집중하되, **고객이 원하고, 만들 역량이 있고, 사는 것보다 나은 기능부터** 시작하라."
